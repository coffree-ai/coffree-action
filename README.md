# coffree-action

GitHub Actions composite action that drives a Coffree coding job through
its full server-side lifecycle. It downloads the coffree CLI binary,
installs `gh` / `gitleaks` if missing, exchanges a GHA OIDC token for a
job authentication token, configures the gh / git credential shims,
runs `coffree run-job`, and cleans the shims up in a post step.

The agent / orchestrator logic lives entirely in the
[coffree-ai/coffree](https://github.com/coffree-ai/coffree) `coffree`
CLI; this action is the GHA-side launcher that supplies it with the
credentials and shims defined in
[`docs/specs/job-execution/gha-runtime.md`](https://github.com/coffree-ai/coffree/blob/main/docs/specs/job-execution/gha-runtime.md)
§3-§4 and
[`docs/specs/job-execution/credential-isolation.md`](https://github.com/coffree-ai/coffree/blob/main/docs/specs/job-execution/credential-isolation.md)
§2-§5.

## Usage

The standard caller workflow is generated by `coffree init` (see
[gha-runtime.md §2.5](https://github.com/coffree-ai/coffree/blob/main/docs/specs/job-execution/gha-runtime.md#L167)).
A condensed copy:

```yaml
name: coffree

on:
  workflow_dispatch:
    inputs:
      job_id:
        description: 'Job ID'
        required: true

permissions:
  contents: write       # git push
  pull-requests: write  # PR create / update
  issues: write         # issue actions
  id-token: write       # OIDC token (required for /auth/exchange-token)

jobs:
  run:
    runs-on: [self-hosted, coffree, coding]
    timeout-minutes: 360  # 6 hours (GHA maximum)

    services:
      api-proxy:
        image: ghcr.io/coffree-ai/api-proxy:v1
        ports:
          - 3100:3100
        volumes:
          - ${{ runner.temp }}/coffree-tokens:/tokens
        env:
          ANTHROPIC_API_KEY: ${{ secrets.COFFREE_ANTHROPIC_API_KEY }}
          OPENAI_API_KEY: ${{ secrets.COFFREE_OPENAI_API_KEY }}
          GEMINI_API_KEY: ${{ secrets.COFFREE_GEMINI_API_KEY }}
          GITHUB_TOKEN: ${{ github.token }}

    steps:
      - uses: actions/checkout@v4
        with:
          persist-credentials: false

      - uses: coffree-ai/coffree-action/coding@v1
        with:
          job_id: ${{ github.event.inputs.job_id }}
          server_url: https://coffree.example.com
          # Local LLM / OpenAI-compatible endpoint (omit for SaaS-only setups).
          llm_base_url: ${{ vars.COFFREE_LLM_BASE_URL }}
          llm_api_key: ${{ secrets.COFFREE_LLM_API_KEY }}
```

`server_url` must be scheme + host without a trailing slash. It is used
both as the `/auth/exchange-token` endpoint and as the OIDC `audience`
claim, so the value must equal the server's `PUBLIC_URL` exactly.

`persist-credentials: false` is mandatory: it stops `actions/checkout`
from leaving the GitHub Token in `.git/config`, ensuring the agent only
sees credentials through the shim path.

## Inputs

| Input | Required | Description |
| --- | --- | --- |
| `job_id` | yes | Job ID issued by the coffree server (passed in via `workflow_dispatch.inputs.job_id`). |
| `server_url` | yes | `scheme://host` of the coffree deployment, no trailing slash. Used for both the exchange-token endpoint and the OIDC audience. |
| `llm_base_url` | no | URL for the local LLM / OpenAI-compatible endpoint (`provider = ollama` / `openai-compatible`). Empty disables the local route. |
| `llm_api_key` | no | Credential for the local LLM / OpenAI-compatible endpoint (e.g. OpenRouter). Ollama / vLLM single-node deployments may leave this blank. |

The `services.api-proxy` container generates the API proxy / credential
proxy authentication tokens at startup and writes them to a shared
volume, so callers do not register them as Secrets. Only AI provider
keys + the optional local-LLM credential need GitHub Secrets entries.

## Composite action steps

1. **Acquire proxy tokens**: read `tokens.json` + `capabilities.json`
   from `${{ runner.temp }}/coffree-tokens/`, mask the proxy tokens, and
   delete both files. If `capabilities.json` lists at least one SaaS
   provider the action injects `COFFREE_AI_PROXY_TOKEN` /
   `COFFREE_AI_PROXY_URL` / `COFFREE_AI_PROXY_PROVIDERS` via
   `$GITHUB_ENV`. The credential proxy token is staged on disk only —
   it never reaches `$GITHUB_ENV`, so the agent process cannot see it.
2. **Download coffree CLI**: detect the runner architecture
   (`linux-x64` / `linux-arm64`), fetch the binary + `sha256` from
   `${server_url}/orchestrator/cli/download/{target}`, verify the
   checksum, and prepend the install dir to `$GITHUB_PATH`.
3. **Install AI agent CLIs**: install `gh` / `gitleaks` from upstream
   tarballs into `$HOME/.local/bin` if they are not already on `PATH`.
   GitHub-hosted runners typically have `gh` pre-installed; this step
   targets self-hosted runners.
4. **Exchange OIDC token**: request a GHA OIDC token (audience =
   `server_url`) and `POST` it to `${server_url}/auth/exchange-token`
   to obtain a `job_token`. The exchange retries up to 3 times after
   the initial attempt on `500` / `503` (4 attempts total, 1s/2s/4s
   backoff) and exits immediately on `400` / `401` / `403` / `404` /
   `411` / `413`. On success the `job_token` is masked and exported via
   `$GITHUB_ENV`.
5. **Configure credential shims**: install a `gh` shim and a
   `git-credential-coffree` helper into a `mktemp -d` directory under
   `$RUNNER_TEMP`, with the credential proxy URL + token embedded in
   the script bodies. The shim dir is prepended to `$GITHUB_PATH` and
   `git config --global credential.helper` is set to the helper. The
   `gh` shim enforces a subcommand allowlist (`pr` / `repo` / `api` /
   `issue`).
6. **Run coffree**: `exec coffree run-job`. All configuration arrives
   via environment variables (`COFFREE_SERVER_URL` / `COFFREE_JOB_TOKEN`
   / `COFFREE_JOB_ID` / optional `COFFREE_AI_PROXY_*` / optional
   `LLM_BASE_URL` + `LLM_API_KEY`); there are no command-line arguments.
7. **Cleanup credential shims** (post step, `if: always()`): remove the
   shim directory and unset `git config --global credential.helper`,
   even when an earlier step failed.

## Error handling for the OIDC exchange

- `500` / `503` from `/auth/exchange-token`: retried up to 3 times after
  the initial attempt (4 attempts in total, 1s / 2s / 4s backoff), then
  the step fails with the response body printed.
- `400` / `401` / `403` / `404` / `411` / `413`: non-retryable, the step
  exits immediately with the response body printed.
- Missing `ACTIONS_ID_TOKEN_REQUEST_URL` or
  `ACTIONS_ID_TOKEN_REQUEST_TOKEN` (i.e. the workflow forgot
  `permissions: id-token: write`): the step exits with a directive
  error message.

## Versioning
