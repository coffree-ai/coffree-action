# coffree-action

GitHub Actions composite action that drives a Coffree job through its server-
side lifecycle.

This repo currently ships **only a dummy runner** (`coding/action.yml`) that
calls the four HTTP endpoints (`render` -> `progress` -> `heartbeat` ->
`complete`) in sequence with `curl`. No real agent is started; the action
exists to give the server-side dispatch / progress / heartbeat / complete
plumbing in [coffree-ai/coffree](https://github.com/coffree-ai/coffree) a
caller it can observe end-to-end.

The production runtime (TypeScript binary, agent SDK adapters, credential
proxy / shims, OIDC token exchange) lands in subsequent slices of the same
phase. Once it ships this action will grow the steps documented in
`docs/specs/job-execution/gha-runtime.md` (Acquire proxy tokens, Download
CLI, Install agents, Exchange OIDC token, Configure shims, Run coffree).

## Usage (dummy runner)

```yaml
jobs:
  coffree-coding:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
        with:
          persist-credentials: false
      - uses: coffree-ai/coffree-action/coding@v0
        with:
          job_id: ${{ inputs.job_id }}
          server_url: https://coffree.example.com
          job_token: ${{ secrets.COFFREE_JOB_TOKEN }}
```

The `permissions:` block above is the future-compatible set: it matches the
generated workflow shape from `coffree init` (gha-runtime.md §2.2) so callers
do not have to widen permissions when the production runtime lands. The
dummy runner itself only needs network access to `server_url`; it does not
push commits, open PRs, comment on issues, or exchange OIDC tokens. The
`id-token: write` permission becomes load-bearing once OIDC token exchange
ships in a later P4.x slice.

`server_url` must be scheme + host without a trailing slash (e.g.
`https://coffree.example.com`). `job_token` is registered as a log mask in
the first step so subsequent log lines that contain it are scrubbed.

In production the workflow file generated by `coffree init` exchanges a GHA
OIDC token for `job_token` per
[`docs/specs/auth/overview.md`](https://github.com/coffree-ai/coffree/blob/main/docs/specs/auth/overview.md)
runner authentication. Until that exchange step ships, callers pass the token
directly per gha-runtime §4.3.
