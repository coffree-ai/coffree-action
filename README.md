# coffree-action

GitHub Actions composite action that drives a Coffree job through its server-
side lifecycle.

This repo currently ships **only a dummy runner** (`coding/action.yml`) that
calls the four HTTP endpoints (`render` -> `progress` -> `heartbeat` ->
`complete`) in sequence with `curl`. No real agent is started; the action
exists to give the server-side dispatch / progress / heartbeat / complete
plumbing in [coffree-ai/coffree](https://github.com/coffree-ai/coffree) a
caller it can observe end-to-end.

The production runtime (TypeScript binary, agent SDK adapters, credential
proxy / shims) lands in subsequent slices of the same phase. Once it ships
this action will grow the steps documented in
`docs/specs/job-execution/gha-runtime.md` (Acquire proxy tokens, Download
CLI, Install agents, Configure shims, Run coffree). The OIDC token exchange
step (gha-runtime.md §4.4) is already wired in this action.

## Usage (production: OIDC exchange)

```yaml
jobs:
  coffree-coding:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
        with:
          persist-credentials: false
      - uses: coffree-ai/coffree-action/coding@v0
        with:
          job_id: ${{ inputs.job_id }}
          server_url: https://coffree.example.com
```

When `job_token` is omitted the action exchanges a GHA OIDC token at
`${server_url}/auth/exchange-token` per
[`docs/specs/auth/overview.md`](https://github.com/coffree-ai/coffree/blob/main/docs/specs/auth/overview.md)
runner authentication / [`gha-runtime.md` §4.4](https://github.com/coffree-ai/coffree/blob/main/docs/specs/job-execution/gha-runtime.md).
`permissions: id-token: write` is required for that exchange; the rest of
the permission set matches the workflow shape generated by `coffree init`
(gha-runtime.md §2.2) so callers do not have to widen permissions when the
production runtime lands. The dummy runner itself only needs network access
to `server_url` and the GHA OIDC provider; it does not push commits, open
PRs, or comment on issues.

`server_url` must be scheme + host without a trailing slash (e.g.
`https://coffree.example.com`). It is used both as the exchange-token
endpoint host and as the OIDC `audience` claim, so the value must match the
server's `PUBLIC_URL` exactly. The exchanged `job_token` is registered as a
log mask before being published as `COFFREE_JOB_TOKEN` for the downstream
steps.

## Usage (test / development: pre-issued token)

```yaml
      - uses: coffree-ai/coffree-action/coding@v0
        with:
          job_id: ${{ inputs.job_id }}
          server_url: https://coffree.example.com
          job_token: ${{ secrets.COFFREE_JOB_TOKEN }}
```

Pass `job_token` directly to skip OIDC exchange (gha-runtime.md §4.3). Use
this only for tests where minting a fresh token through the server is
inconvenient; production should leave `job_token` empty.

## Error handling for OIDC exchange

Mirrors the table in
[`gha-runtime.md` §4.4](https://github.com/coffree-ai/coffree/blob/main/docs/specs/job-execution/gha-runtime.md):

- `500` / `503` from `/auth/exchange-token`: retried up to 3 times with
  exponential backoff (1s, 2s, 4s) before failing the step.
- `400` / `401` / `403` / `404` / `411` / `413`: non-retryable, the step
  exits immediately with the response body printed.
- Missing `ACTIONS_ID_TOKEN_REQUEST_URL` / `ACTIONS_ID_TOKEN_REQUEST_TOKEN`
  (i.e. the workflow forgot `permissions: id-token: write`): the step exits
  with a directive error message.
